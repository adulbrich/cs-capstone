# Interest-Based Project Discovery with Bedrock Embeddings

Date: 2026-07-22
Status: Approved design, ready for implementation planning

## Summary

Project discovery today is keyword driven: a Postgres `tsvector` with a GIN
index, queried through `websearch_to_tsquery` in `searchProjectsImpl`. A
student who is interested in "robotics" will not find a project titled
"Autonomous rover telemetry pipeline" unless the word "robotics" happens to
appear in it. The README roadmap asks to "allow users to write their interests
etc and be presented with relevant projects".

This design adds semantic matching. A user writes a free-text description of
their interests on their profile. That text is embedded with Amazon Titan Text
Embeddings V2 through the Bedrock client the app already uses for AI review.
Published projects are embedded the same way. A new `sort=recommended` option
on `/projects` orders results by cosine distance between the two vectors,
inside the existing filter set.

Projects are embedded **only when published**, and re-embedded when a published
project's indexed text changes. Drafts, submitted, and rejected projects are
never sent to Bedrock.

Keyword search is not replaced. The search box, its operators, and
`ts_rank` relevance ordering all stay exactly as they are. Embeddings add a
third sort mode alongside them, and the existing full-text index doubles as the
graceful-degradation path when a vector is unavailable.

## Goals

- Let a signed-in user describe their interests once, in their own words, and
  get projects ranked by semantic similarity to that description.
- Embed a project only at publish time, and keep its vector current when the
  published text changes.
- Never let an embedding failure block a publish, a save, or a page load.
- Keep every filter, the search box, and pagination working unchanged under the
  new sort mode, with all state in URL search params as the app requires.
- Keep the whole pipeline testable without an AWS call.

## Non-goals (out of scope for this release)

- Replacing or re-weighting full-text search. `search_vector` and
  `websearch_to_tsquery` are untouched.
- Semantic search from the search box. The box stays lexical; semantics are
  reached through the sort control.
- Embedding anything other than projects and user interests. No comments, no
  inventory, no categories.
- Recommendation emails, digests, or notifications when new matching projects
  publish. Noted in Future work.
- Chunking long projects into multiple vectors. One project is one vector.
- Cross-user or collaborative signals (bookmarks, views, "students like you").
  Only the user's own written interests drive ranking.

## Decisions

- **Bedrock embeddings over the tsvector-only alternative (user decision).**
  Semantic match is the point of the feature; lexical overlap is what already
  exists and does not deliver it.
- **`amazon.titan-embed-text-v2:0`, 1024 dimensions, normalized.** 8,192-token
  (50,000-character) input limit, which comfortably holds a whole project.
  Dimensions are configurable (1024 default, 512, 256) but fixed at 1024 here;
  the value is captured in the source hash so a future change invalidates
  cleanly. Normalization at the API makes cosine distance and inner product
  equivalent and keeps distances comparable across rows.
- **A separate `user_interests` table, not columns on `user`.** `user` is
  generated by the Better Auth CLI into `src/db/auth-schema.ts`, which the
  README states must never be hand-edited; only `additionalFields` survive
  regeneration, and Better Auth has no `vector` type. A separate table owned by
  the app is the only safe home. `projects` is app-owned, so its vector is a
  plain column.
- **The vector column lives on `projects`, not in a side table.** It is 1:1
  with the row, written in the same code path, and read in the same query. A
  join would buy nothing.
- **Embed on publish, re-embed on save when published (user decision).** The
  trigger is the publish event, so drafts are never embedded. A source hash
  detects when a published project's indexed text has changed and triggers a
  refresh on save. A project rewritten after publishing therefore does not keep
  matching on its old text.
- **The hash covers the source text, the model id, and the dimensions.**
  Changing the embedding model invalidates every vector automatically; the
  backfill sweep then refreshes them without a manual reset.
- **The embedding call is awaited inside the request, not fire-and-forget
  (user decision).** An unawaited promise on Fargate can be killed when the
  task rolls, silently losing the vector. The call adds roughly 200-400ms to a
  publish or a save of a published project. If that is noticeable in practice,
  the answer is a pending indicator on the submitting button, not a change in
  the write path.
- **Interest vectors are computed on profile save, synchronously.** The user
  just asked for recommendations and expects them to work on the next page
  load.
- **Category and program preferences are hard filters, not blended scores.**
  Ranking is pure vector distance within the already-filtered set. Blending a
  cosine distance with a categorical overlap count requires tuning weights with
  no data to tune them against.
- **Null-embedding projects sort last, they are not dropped.** A project that
  failed to embed, or predates the feature, must remain reachable under every
  sort mode.

## Schema changes

One migration. Because `drizzle-kit generate` does not emit extension
statements, the generated SQL is hand-edited to put the extension first.

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

`projects` gains:

- `embedding vector(1024)` (nullable)
- `embedding_source_hash text` (nullable)
- `embedding_updated_at timestamptz` (nullable)

New table `user_interests`:

- `user_id text` primary key, FK to `user.id`, `ON DELETE CASCADE`
- `interests_text text not null` (application limit 2,000 characters)
- `embedding vector(1024)` (nullable)
- `embedding_source_hash text` (nullable)
- `embedding_updated_at timestamptz` (nullable)
- `created_at`, `updated_at`

Indexes:

- `CREATE INDEX ON projects USING hnsw (embedding vector_cosine_ops)`.

At the expected scale (hundreds of published projects) a sequential scan would
be fine, and an HNSW index over a filtered query can even be skipped by the
planner. It is included because it is cheap to create now and avoids a second
migration later. It is explicitly not a correctness dependency.

Drizzle exposes both pieces natively: `vector("embedding", { dimensions: 1024 })`
from `drizzle-orm/pg-core`, and `.using("hnsw", table.embedding.op("vector_cosine_ops"))`
on the index builder.

### Environment support

- Production: RDS PostgreSQL 18 supports pgvector 0.8.1, so the extension
  statement is all that is required. `infra/iam.tf:57-61` already grants
  `bedrock:InvokeModel` on `*`, so no IAM change is needed.
- Local: `docker-compose.yml` changes image `postgres:18` to
  `pgvector/pgvector:pg18`. Same major version, so the existing
  `postgres_data` volume carries over. The compose file currently mounts the
  volume at `/var/lib/postgresql` rather than `/var/lib/postgresql/data`;
  implementation must confirm this still matches the pgvector image's `PGDATA`
  and adjust the mount if it does not.
- Bedrock model access for Titan Text Embeddings V2 must be granted in the
  target account and region. This is an account-level toggle, separate from
  IAM, and is a plausible first-deploy failure. It belongs in `DEPLOYMENT.md`.

## Embedding pipeline

### `src/lib/_internal/bedrock-embed.ts`

Mirrors the injection pattern already used for AI review, where
`project-review-core.ts:115` takes `invoke: ConverseFn = bedrockConverse` as a
default parameter so tests substitute a fake.

```ts
export type EmbedFn = (text: string) => Promise<number[]>;
export const bedrockEmbed: EmbedFn = ...;
```

`bedrockEmbed` sends an `InvokeModelCommand` with body
`{ inputText, dimensions, normalize: true }` and reads `embedding` from the
response. It reuses `getBedrockClient()` and `buildBedrockConfig()` from
`src/lib/_internal/bedrock.ts`, so credentials resolve through the ECS task
role in production exactly as the review path does. Model id and dimensions
come from `BEDROCK_EMBEDDING_MODEL_ID` and `BEDROCK_EMBEDDING_DIMENSIONS`, with
defaults `amazon.titan-embed-text-v2:0` and `1024`.

### `src/lib/embedding-source.ts`

Pure and dependency-free, so it is importable from both sides and trivially
testable:

- `buildProjectEmbeddingSource(project, categoryNames, programLabel): string`
  concatenates the title, the six long fields, the category names, and the
  program label under labelled headings, then truncates to 45,000 characters
  (under the model's 50,000-character limit, with headroom).
- `buildInterestsEmbeddingSource(interestsText): string`.
- `embeddingHash(source, modelId, dimensions): string`, a SHA-256 hex digest.

### `refreshProjectEmbedding(projectId, embed?)`

In `src/server/_internal/project-embeddings.ts`. Loads the project with its
categories and program, returns early if the project is not `published` or
`deletedAt` is set, builds the source and hash, returns early if the hash
matches the stored one, otherwise embeds and writes `embedding`,
`embedding_source_hash`, and `embedding_updated_at` in a single update.

This is the only function that writes a project vector. Every trigger routes
through it, and it is idempotent.

**How the fake reaches it in tests.** `refreshProjectEmbedding` takes
`embed: EmbedFn = bedrockEmbed` as a trailing default parameter, and the
transition and update helpers thread an optional `embed` through to it in the
same position. The existing `*As(viewer, ...)` test seams therefore accept a
fake and pass it down, which is what keeps the integration suite free of AWS
calls. Threading the parameter explicitly is preferred over a module-level
mutable default, because the integration tests share a single fork and a
swapped global would leak between cases.

### Trigger points

- `performTransitionAs` and `forceTransitionAs` in
  `src/server/_internal/projects.ts`, when the target status is `published`.
  Both paths reach `published` and both must be hooked.
- The project update path, after a successful save, when the project's status
  is `published`. The hash check makes this a no-op for edits that do not touch
  indexed text.
- Profile save writes the user's interest vector through the equivalent
  `refreshInterestsEmbedding(userId, embed?)`.

### Failure semantics

Three named behaviours, not "async":

1. **The transaction commits first.** `refreshProjectEmbedding` is called
   *after* the existing `db.transaction` block returns. Embedding is never part
   of that transaction and can never roll back a publish, a status history row,
   or a notification.
2. **Failure is tolerated and logged, never surfaced as a request failure.**
   The call is awaited but wrapped, so a Bedrock outage, a throttle, or a
   missing model grant leaves `embedding` null or stale and the publish still
   succeeds. The user is not shown an error, because nothing they asked for
   failed.
3. **A backfill sweep is the safety net.** `npm run embeddings:backfill` runs
   `scripts/backfill-embeddings.ts`, which selects published, non-deleted
   projects whose `embedding` is null or whose stored hash differs from the
   freshly computed one, and refreshes them serially with a small delay to stay
   under Bedrock throttling limits. It is idempotent and safe to re-run. It
   serves three purposes: the initial backfill of already-published projects,
   recovery from transient failures, and re-embedding after a model change.

Profile save differs on point 2: an interests embedding failure *is* surfaced,
because the user explicitly asked for their recommendations to be built. The
text saves regardless; the UI reports that recommendations could not be
prepared and offers a retry.

## Server functions

- `getMyInterests` (GET): returns `{ interestsText, hasEmbedding }` for the
  current user. Never returns the vector.
- `saveMyInterests` (POST): validates `interestsText` (max 2,000 characters),
  upserts the row, refreshes the embedding, returns
  `{ saved: true, embedded: boolean }`.
- `searchProjects` (existing, GET): its input schema gains
  `sort: z.enum(["relevance", "newest", "recommended"]).default("relevance")`.
  The same field is added to `validateSearch` on the `/projects` route so the
  sort is a shareable URL search param like every other filter.

**The default must be `relevance`, not `newest`.** Ordering is implicit today:
`searchProjectsImpl:31-33` uses `ts_rank` when a query string is present and
`publishedAt DESC` otherwise. Defaulting an explicit param to `newest` would
override that conditional and silently reorder every keyword search by date.
`relevance` reproduces current behaviour exactly.

`searchProjectsImpl` changes only in its ordering branch. All existing
conditions (status, `deletedAt`, program, categories, full-text match) are
untouched and continue to apply.

- `relevance` (default): `ts_rank(...)` when a query is present, otherwise
  `desc(projects.publishedAt)`. This is the existing behaviour, unchanged.
- `newest`: always `desc(projects.publishedAt)`. New capability: it lets a user
  sort search results by date, which was not previously possible.
- `recommended`, viewer has a vector:
  `ORDER BY projects.embedding IS NULL, projects.embedding <=> $userVector`.
  The null-first boolean pushes unembedded projects to the end.
- `recommended`, viewer has interests text but no vector (Bedrock failed):
  falls back to `relevance` ordering, and the UI reports that recommendations
  are not ready yet with a retry link to the profile.

  **Do not fall back to full-text ranking of the interests text.**
  `websearch_to_tsquery` ANDs unquoted terms, so a paragraph of interests
  becomes a conjunction of every lexeme in it and matches nothing. The result
  would be an empty list presented as "ranked by your interests", which reads
  to the user as "no project suits you". An OR-joined query built from
  `tsvector_to_array(to_tsvector(...))` would work, but it is real complexity
  on a path that should be rare and self-healing: the profile save already
  surfaces the failure and offers a retry, and the backfill sweep repairs
  project-side gaps. Falling back to the ordinary ordering and saying so is
  both simpler and more honest.
- `recommended`, viewer has no interests at all: falls back to `relevance`. The
  UI prevents reaching this state, but the server must not depend on that.

The viewer's vector is loaded server-side from `user_interests` on the same
request. It is never sent to the client.

## Frontend

### Profile

A new "Your interests" section on `/profile`: a labelled textarea with a
character counter, a short explanation of what the text is used for and who can
see it (only the user), and a link to `/projects?sort=recommended` after a
successful save. Placeholder copy should prompt for substance, for example
"Robotics, embedded systems, and anything involving sensor data. I have taken
CS 344 and I am comfortable with C and Python."

### Projects listing

A `Sort` select joins the filter bar, as a URL search param like every other
filter. The `Recommended for you` option is disabled, with a link to
`/profile`, when the user is signed out or has not written interests.

When `sort=recommended` is active, a single line under the filter bar reads
"Ranked by your interests" with an edit link. A per-result match explanation is
deliberately omitted: with pure vector similarity there is no term overlap to
show, and a distance-derived label would be a number dressed as a reason.

### Publish and save

Publish, approve, and save actions on a published project now include a Bedrock
round trip. The submitting button uses the form's existing pending state to show
a spinner and disable itself for the duration, which is the same treatment other
slow mutations already receive. No new mechanism, and no user-facing mention of
embeddings.

## Auth gating and privacy

- Interests text is written and read only by its owner. There is no admin view,
  no staff view, and no listing endpoint that exposes it.
- The vector never leaves the server, under any code path.
- `sort=recommended` requires a session. Signed-out users get the `newest`
  ordering and the disabled option.
- Deleting a user cascades the `user_interests` row.

## Configuration

New environment variables, added to `.env.example`, `infra/variables.tf`, and
the ECS task definition in `infra/ecs.tf` alongside the existing
`BEDROCK_MODEL_ID`:

- `BEDROCK_EMBEDDING_MODEL_ID` (default `amazon.titan-embed-text-v2:0`)
- `BEDROCK_EMBEDDING_DIMENSIONS` (default `1024`)

`BEDROCK_REGION` is shared with the existing review feature and must host both
models.

## Testing

Every test injects a fake `EmbedFn`. No test calls AWS.

- Unit: `buildProjectEmbeddingSource` includes each field, includes category
  and program labels, and truncates at the limit.
- Unit: `embeddingHash` is stable for identical input and changes when the
  model id or the dimension count changes.
- Unit: `refreshProjectEmbedding` is a no-op for a draft, a no-op when the hash
  is unchanged, and writes all three columns when the hash differs.
- Integration: publishing a project populates `embedding`; the project is
  published and the status history row exists even when the injected `EmbedFn`
  throws.
- Integration: saving an already-published project with a changed problem
  statement updates the vector; saving with only a `notes` change does not.
- Integration: saving a draft never calls the injected `EmbedFn`.
- Integration: `sort=recommended` orders by distance using hand-written unit
  vectors, keeps program and category filters applied, and places a
  null-embedding project last.
- Integration: a user with interests text but no vector gets the `ts_rank`
  fallback ordering rather than an error.
- Integration: deleting a user removes the `user_interests` row.

## Risk callouts

- **pgvector availability is the hard dependency.** Verified for both targets:
  RDS PostgreSQL 18 supports pgvector 0.8.1, and `pgvector/pgvector:pg18`
  exists on Docker Hub. If the extension cannot be created in production, the
  migration fails and nothing else in this design can ship.
- **Bedrock model access is an account-level grant.** Titan Text Embeddings V2
  must be enabled separately from IAM permissions. The likely failure mode is a
  successful deploy where every project silently fails to embed, which the
  backfill script's output makes visible.
- **Embedding quality is unmeasured.** There is no relevance benchmark and no
  way to build one before the feature exists. The first real evaluation is
  qualitative, against real projects, after the initial backfill.
- **Latency is inside the write path by choice.** If a publish becomes
  noticeably slow, the escape hatch is moving `refreshProjectEmbedding` to a
  queue or a scheduled sweep. The design deliberately puts all embedding writes
  behind one function so that move is local.
